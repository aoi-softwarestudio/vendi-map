# 🔌 Agent Skill: Systematic API Integration, Proxy Setup & Secure Token Management

## Purpose
このスキルは、開発部門エージェントが、LLM（Gemini等）、地図API（OSM/Nominatim等）、決済ゲートウェイなどの外部APIを安全かつ堅牢にシステムへ統合し、ネットワーク遅延や認証エラーに対する耐障害性の高い中継プロキシサーバーの設計、環境変数の安全なロード、および接続テストを自動化するための標準手順を定義します。

---

## 📋 Execution Checklist

### Phase 1: API要件定義とプロトタイプ接続テスト
1. **API仕様の監査**:
   - 利用するAPIのエンドポイント、リクエスト/レスポンス形式（JSON等）、認証方式（APIキー、OAuth2、ベアラートークン）を整理します。
2. **疎通確認スクリプトの作成**:
   - `urllib.request` や `requests` を用いて、最小限の疎通確認を行うテストコードを `scratch` ディレクトリ等に作成して実行します。

### Phase 2: 安全な認証管理とプロキシ設計
1. **環境変数の切り出しと厳重管理**:
   - APIキーなどの秘密情報をコード内にハードコーディングせず、必ず `.env` もしくはシステムの環境変数から動的にロード（`os.getenv` 等）する設計にします。
2. **中継プロキシサーバーの実装**:
   - フロントエンドからAPIキーを直接露出させないため、FastAPI（`server.py`）等に中継用プロキシエンドポイント（例: `/api/gemini-proxy`）を設置します。
   - プロキシ内でAPIキーをヘッダーにインジェクションし、レスポンスのパースエラーをハンドリングします。

### Phase 3: 耐障害性設計と疎通エラー処理
1. **タイムアウトとリトライ処理**:
   - 外部API呼び出しのハングアップを防ぐため、適切なタイムアウト値（例: `timeout=10`）を設定します。
   - 一時的なネットワークエラーに対して、指数バックオフ（Exponential Backoff）アルゴリズムを用いた自動再試行ロジックを組み込みます。
2. **フォールバック挙動の定義**:
   - API制限やサーバー障害時に、アプリ全体がクラッシュするのを防ぐため、ローカル生成の模擬データや静的キャッシュを返すスマート・フォールバックを実装します。

---

## 📝 Deliverable Format (`api_integration_report.md`)
API連携の設計と接続検証時に作成する報告書のフォーマットです。

```markdown
# 🔌 API統合・セキュリティ接続接続報告書

## 1. 接続ターゲットAPI概要
* **API名**: [例: Gemini Pro API]
* **目的**: [例: 自動チャット生成および要約]
* **認証タイプ**: [例: Bearer Token / API Key]

## 2. 実装およびセキュリティ設計
* **環境変数名**: `GEMINI_API_KEY` (ローカルハードコードなし)
* **プロキシエンドポイント**: `/api/gemini-proxy` (フロントエンド隠蔽完了)
* **エラー時のフォールバック**: [例: APIエラー時はローカルのルールベース回答を返却]

## 3. 疎通テスト結果
| テスト項目 | 期待される応答 | 実際の応答時間 | ステータス |
| :--- | :--- | :--- | :--- |
| 正常リクエスト接続 | Status 200 (Valid JSON) | 120ms | 🟢 SUCCESS |
| タイムアウト検知 | TimeoutException (30s) | 30000ms | 🟢 HANDLED |
| 無効トークン拒否 | Status 401 (Unauthorized)| 85ms | 🟢 HANDLED |
```
